Why llm-gateway

Two very different billing models, hard-won Windows fixes, one honest contract.

💳

One cost contract, two billing models

API calls cost real money per token. CLI agents run against a subscription you already pay for. Mixing them up in a dashboard is a real bug we hit — this library keeps them distinguishable from the source.

🛠️

Windows fixes, done once

.cmd shims, cmd.exe mangling multi-line prompts, Codex's read-only sandbox by default — all fixed here instead of rediscovered per project.

⏱️

No surprise hangs

Every subprocess call has a 15-minute timeout and raises CliAgentError instead of hanging forever or failing silently.

📦

Library, not a service

No server, no config file, no daemon. Just async functions you await and a Pydantic model you read.

The TokenUsage contract

Every function returns (text, TokenUsage) — aggregate across calls with plain arithmetic, splitting by billing when you display totals.

fieldmeaning
stage"extractor", "agent_plan", "agent_exec_codex", ...
modelmodel name used for the call
prompt_tokens / completion_tokenstoken counts, when the provider reports them
cost_usdreal charge (api) or cost-equivalent (subscription)
billing"api" (real charge) | "subscription" (cost-equivalent)
task_id / member_idoptional — attribute cost to your own domain objects

cwd is not a sandbox

⚠ Read before you point cwd at a real project

run_claude_code and run_codex launch a real, fully capable coding agent as a subprocess. cwd sets its working directory — it is not a filesystem jail.

Confirmed by direct testing: the agent can read and write anywhere the OS user account has access, including outside cwd — e.g. following an editable pip install to a sibling project, or triggering git's own parent-directory search for .git and committing to a repo above cwd.

If you need the agent confined to a directory, that requires OS-level isolation (a container, a restricted user account, a VM) — this library does not provide it.
